# AI电商自动化平台工程实施计划

> 给 AI 协作工具使用：执行本计划时，优先使用 Superpowers 的分步骤执行能力。每个任务都要按任务卡、验收标准、审核记录推进，不允许只在聊天里说完成。

## 目标

在 2026-06-01 到 2026-08-07 的 8-10 周内，完成一套可内部试点、可向客户演示、可继续商用交付的 AI 电商自动化平台最小可用版本（MVP）。

## 架构说明

GitHub 是代码、任务卡（Issue）、拉取请求（PR）、版本记录和节点审核的主控中心。Obsidian 只打开仓库里的 `/docs` 文件夹，用来保存知识库、业务决策、接口清单、数据库模型、交付资料和审核结论。

产品形态是一套多客户平台：电脑端后台、手机网页/可安装网页应用（H5/PWA）、飞书协同输出、AI 日报、平台接口同步和后续 ERP 能力预留。

## GitHub协作流程

1. `main` 是主分支，只接受审核通过的拉取请求（PR）。
2. 每张任务卡（Issue）必须写清楚负责人、截止日期、验收标准、模块和风险等级。
3. 每个拉取请求（PR）必须关联任务卡（Issue）。
4. 涉及数据库、权限、同步、客户可见页面、AI 回复、飞书导出、邮件发送的改动，至少 2 人审核。
5. 代码负责人配置（CODEOWNERS）用于后续自动指定审核人，真实 GitHub 账号未确定前先用占位角色。

## Obsidian知识库规则

1. Obsidian 只打开 `/docs` 文件夹。
2. 业务决策、接口权限、数据库模型、风险记录、节点审核结论都写进 Obsidian。
3. Obsidian 不替代 GitHub 任务卡（Issue）和拉取请求（PR）。
4. AI 审核结论必须落到 `节点审核记录.md`。

## 数据库模型

一期核心表包括：

- tenants：客户公司/租户。
- users：平台用户。
- tenant_members：租户成员关系。
- roles：角色。
- permissions：权限点。
- module_entitlements：模块开通。
- platform_connections：平台授权连接。
- sync_jobs：同步任务定义。
- sync_runs：同步运行记录。
- accounts：短视频/店铺/广告账号。
- content_items：视频/素材记录。
- content_metrics：内容指标快照。
- orders：订单主表。
- order_items：订单明细。
- shipments：物流记录。
- customers：客户资料。
- ai_outputs：AI 输出。
- feishu_exports：飞书同步记录。
- tasks：系统待办。
- audit_logs：审计日志。

详细字段以 [[04-数据库模型/一期数据库模型|一期数据库模型]] 为准。

## 平台接口

| 平台 | 一期目标 | 风险 |
|---|---|---|
| 飞书 | 多维表格、群通知、日报输出 | 权限和敏感字段 |
| 抖音 | 企业号、短视频数据、评论/私信预留 | 平台授权审核 |
| 抖店 | 订单、售后、物流模型预留 | 接口权限和店铺授权 |
| 独立站/Shopify | 订单、客户、支付、Webhook | 站点接口差异 |
| TikTok/TK | 账号内容、视频指标 | 官方接口能力限制 |
| 邮件 | 订单通知、节日祝福、达人建联 | 群发风险和退订机制 |

## 节点审核节奏

1. 每日 10 分钟：昨天完成什么、今天做什么、是否阻塞。
2. 周五节点审核：进度、业务方向、代码质量、数据正确性、权限风险、下周调整。
3. 关键拉取请求（PR）审核：数据库、权限、同步、AI 回复、客户可见页面、飞书/邮件输出。

## AI工具规则

1. AI 可以辅助写任务卡、文档、测试、审核意见和非关键脚手架。
2. AI 不允许直接合并拉取请求（PR）。
3. AI 不允许接触生产密钥。
4. AI 不允许直接发送客户邮件、自动退款、自动通过高风险客服回复。
5. Claude/Codex 输出必须由人工负责人确认后才能采纳。
