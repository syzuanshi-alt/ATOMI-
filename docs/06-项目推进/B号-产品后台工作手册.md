# B号-产品后台工作手册

## 角色定位

B 号负责产品后台与手机端体验。你的任务是让客户、运营、客服、老板都能看懂并使用这个系统。你不需要先追求复杂动效，优先保证流程清楚、信息层级清楚、关键操作不容易点错。

你每周必须优先回答三个问题：

1. 客户进入后台第一眼能不能看懂业务状态？
2. 运营和客服每天要处理的任务是否能在 3 步内找到？
3. 手机端是否能完成查看日报、处理待办、审批 AI 建议这些轻操作？

## 你负责的边界

| 负责 | 不负责 |
|---|---|
| 电脑端后台、平台总控、客户后台、手机端 H5/PWA、任务中心、页面流程 | 数据库迁移、第三方授权、真实邮件发送 |
| 页面信息架构、原型、组件状态、空状态、错误状态、移动端验收 | 绕过权限直接展示数据 |

## 用户角色视角

| 用户 | 主要诉求 | B 号需要保证 |
|---|---|---|
| 平台老板/管理员 | 看客户运行情况、异常、交付状态 | 总控清晰、有风险提示 |
| 客户老板 | 看销售、订单、客服、投流、AI 建议 | 指标少而准，不堆无效图 |
| 运营 | 看账号数据、订单异常、素材效果、达人跟进 | 筛选和任务流顺手 |
| 客服 | 看客户信息、会话摘要、AI 建议回复 | 操作入口明确，有人工确认 |
| 项目负责人 | 看节点、风险、交付材料 | 审核页面和文档链接清楚 |

## 第 1 周详细步骤

### B-01 绘制平台总控页面结构

1. 阅读 `业务目标.md`、`平台总控.md`、`客户后台.md`。
2. 先画结构，不急着做美化。
3. 平台总控第一屏建议包含：
   - 今日订单/GMV/客服待处理/同步异常。
   - 客户列表和运行状态。
   - AI 建议待审核数量。
   - 数据同步失败提醒。
   - 本周交付/风险节点。
4. 输出页面结构图或 Markdown 描述。
5. 写清楚哪些模块一期做，哪些模块二期做。

验收标准：

- 老板、运营、客服三类人都能找到自己的入口。
- 不出现堆满图表但看不出重点的页面。
- 页面结构与一期范围一致。

### B-02 客户后台信息架构

1. 拆出客户后台导航：
   - 总览
   - 订单物流
   - 客服中心
   - 内容账号
   - 达人 BD
   - 素材投流
   - AI 日报
   - 飞书/邮件配置
   - 设置
2. 每个导航写 3 个字段：
   - 目标用户
   - 核心信息
   - 一期操作
3. 将不适合一期的页面标记为二期，避免范围膨胀。

验收标准：

- 导航不超过客户能理解的数量。
- 每个页面有明确业务目标。
- 每个页面能对应数据库或接口来源。

## 第 2 周详细步骤

### B-03 手机端轻工作台结构

1. 手机端不复制整个电脑后台，只做轻工作台。
2. 一期手机端只保留：
   - 今日概览
   - 待处理任务
   - AI 日报
   - 订单/物流异常
   - 客服待审核
   - 飞书同步状态
3. 每个手机页面必须适合单手查看，不做复杂表格。
4. 关键操作需要二次确认，例如发送邮件、采纳 AI 回复、标记异常已处理。

验收标准：

- 手机端能完成查看、审批、处理待办。
- 不要求完成复杂配置。
- 每个操作都有权限控制和状态反馈。

## 页面验收原则

1. 每个页面必须有明确主任务。
2. 首页不要做成宣传页，要直接进入业务后台。
3. 卡片、表格、筛选、详情页要能承载真实数据，不只放示例文案。
4. 所有按钮要有状态：默认、加载、成功、失败、无权限。
5. 移动端优先显示待办和异常，不展示复杂配置。
6. 文案要像工具，不像广告。

## GitHub 操作步骤

1. 页面结构先写进 Obsidian，再拆 Issue。
2. 每个页面或组件对应一个 Issue。
3. 分支命名：`feature/B-编号-简短英文名`。
4. PR 必须附截图或录屏路径。
5. 如果页面依赖 A 的字段，必须在 Issue 中写清楚字段名和接口期望。
6. 如果页面需要 C 的飞书/AI/邮件数据，必须写清楚数据来源和空状态。

- [ ] 手机端是否没有照搬大表格？
- [ ] 操作是否有二次确认和错误提示？
- [ ] 是否引用了正确的数据来源？
- [ ] 截图或原型是否已放入 PR/Issue？
