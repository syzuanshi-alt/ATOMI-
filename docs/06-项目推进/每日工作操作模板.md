# 每日工作操作模板

## 每天开始前 10 分钟

每个人每天开工前都按这个顺序做，不要跳：

1. 打开 Obsidian。
2. 进入 `D:\SHKF\docs` 知识库。
3. 打开 [[完整工作计划]]，确认本周目标。
4. 打开自己的手册：
   - A 号打开 [[A号-平台底座工作手册]]
   - B 号打开 [[B号-产品后台工作手册]]
   - C 号打开 [[C号-集成AI交付工作手册]]
5. 打开 GitHub 仓库：`https://github.com/syzuanshi-alt/ATOMI-`
6. 打开 GitHub 项目看板（Project，项目看板）。
7. 找到分配给自己的任务卡（Issue，任务卡）。
8. 看任务卡里 4 个最重要的信息：
   - 今天要做什么。
   - 截止日期是哪天。
   - 验收标准是什么。
   - 有没有前置依赖。

## 开始工作时必须写一条任务卡评论

复制这个模板到自己的 GitHub 任务卡评论区：

```markdown
今日开始：
- 我今天处理：
- 预计产出：
- 需要别人配合：
- 当前阻塞：
```

不会写时，按这个例子：

```markdown
今日开始：
- 我今天处理：B-01 平台总控页面结构。
- 预计产出：一份后台首页结构草案。
- 需要别人配合：A 确认订单、客服、同步异常字段。
- 当前阻塞：暂无。
```

## 做事过程中怎么记录

### 如果你写代码

1. 不要直接改主分支 `main`。
2. 先拉取最新代码：

```powershell
git -C D:\SHKF pull
```

这条命令的意思：把 GitHub 最新内容拉到本地。

3. 新建任务分支：

```powershell
git -C D:\SHKF checkout -b feature/A-01-project-skeleton
```

这条命令的意思：为 A-01 任务开一条单独工作线。

4. 改完后查看变更：

```powershell
git -C D:\SHKF status
```

这条命令的意思：看你改了哪些文件。

5. 提交前先确认没有真实密钥、客户隐私、账号密码。

### 如果你写文档或原型

1. 先在 Obsidian 写到对应页面。
2. 每个页面开头写清楚：
   - 这页给谁看。
   - 解决什么问题。
   - 本周做到什么程度算完成。
3. 文档写完后，把链接贴回 GitHub 任务卡。
4. 如果有截图，写清楚截图文件位置。

### 如果你整理接口或平台资料

1. 不要把真实密钥写进 Obsidian 或 GitHub。
2. 只写字段名、权限名、获取方式、风险。
3. 对每个平台标记：
   - 能不能接接口（API）。
   - 不能接时能不能导出表格。
   - 谁提供账号。
   - 最晚什么时候准备好。

## 每天下班前 15 分钟

每个人必须更新任务卡。

复制这个模板：

```markdown
今日收尾：
- 今天完成：
- 验收证据：
- 明天继续：
- 阻塞问题：
- 需要审核：
```

示例：

```markdown
今日收尾：
- 今天完成：飞书测试应用权限清单初稿。
- 验收证据：已写入 Obsidian《飞书》页面。
- 明天继续：补充多维表格字段。
- 阻塞问题：需要确认飞书管理员账号。
- 需要审核：需要项目负责人确认字段是否包含敏感信息。
```

## 状态怎么改

| 情况 | GitHub 看板状态 |
|---|---|
| 还没开始 | 未开始 |
| 今天正在做 | 进行中 |
| 做完了，等别人看 | 待审核 |
| 已验收通过 | 已完成 |
| 卡住，自己解决不了 | 阻塞 |

阻塞时必须写清楚：

1. 卡在哪里。
2. 需要谁帮忙。
3. 最晚什么时候需要解决。
4. 如果不解决，会影响哪个任务。

## 小白常见错误

1. 只在聊天里说完成，但 GitHub 没更新。
2. 写“已完成”，但没有截图、链接、测试结果。
3. 不看验收标准就开始做。
4. 改了字段不告诉其他人。
5. 把真实账号、密钥、客户信息发到仓库里。
6. 不懂也不问，等到周五才暴露问题。
