# 每周节点审核操作模板

## 周五审核目标

周五审核不是开会聊天，是判断项目有没有跑偏、任务有没有完成、风险有没有暴露、下周计划要不要改。

每周必须回答：

1. 本周做完了吗？
2. 哪些没做完？
3. 为什么没做完？
4. 有没有业务跑偏？
5. 有没有权限、数据、AI、接口、邮件风险？
6. 下周要改什么？

## 周五上午：每个人先自查

每个人在自己任务卡里写：

```markdown
本周自查：
- 本周任务：
- 完成情况：
- 验收证据：
- 未完成原因：
- 风险：
- 下周建议：
```

没有验收证据的任务，不能标记已完成。

## 周五下午：审核顺序

1. 项目负责人打开 GitHub 项目看板。
2. 先看“风险审核”视图。
3. 再看“阻塞”状态。
4. A 号汇报底座、数据库、权限、同步日志。
5. B 号汇报后台、手机端、客户是否看得懂。
6. C 号汇报飞书、接口、AI、邮件、测试资料。
7. 项目负责人记录结论。
8. Claude/Codex 只做辅助审核。
9. 结论写入 [[节点审核记录]]。

## 审核结果写入格式

把结果写入 [[节点审核记录]]：

```markdown

### 本周结论
- 完成情况：
- 是否跑偏：
- 主要风险：
- Claude/Codex 辅助结论：
- 人工最终结论：

### 必须修复
- [ ] 问题：
  - 责任人：
  - 截止日期：

### 下周调整
- 新增：
- 延后：
- 删除：
```

## 审核通过标准

| 类型 | 通过标准 |
|---|---|
| 代码任务 | 有 PR、能运行、有测试或验证截图 |
| 文档任务 | Obsidian 页面已更新，逻辑清楚 |
| 原型任务 | 有页面结构、截图或链接 |
| 接口任务 | 权限、字段、风险、账号准备写清楚 |
| AI 任务 | 有输入范围、提示词、输出、人工审核状态 |
| 客户可见任务 | 文案、截图、流程已经人工检查 |

## 不通过怎么处理

1. 不通过的问题必须变成 GitHub 任务卡。
2. 责任人必须明确。
3. 截止日期必须明确。
4. 如果影响第 2 周计划，当天就调整周计划。
5. 不允许口头说“下周再看”但没有任务卡。
