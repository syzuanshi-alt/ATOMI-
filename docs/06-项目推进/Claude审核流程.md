# Claude审核流程

## 审核定位

Claude/Codex 是第二双眼睛，用来辅助发现业务跑偏、范围膨胀、权限漏洞、AI 风险和测试缺口。它不能替代项目负责人，也不能替代人工验收。

每周五节点审核后，必须把 Claude/Codex 的结论写入 `节点审核记录.md`，不能只停留在聊天窗口。

## 每周审核输入

固定收集以下材料：

1. 本周 GitHub 任务卡（Issue） 列表和状态。
2. 本周 拉取请求（PR） 列表、合并状态、关键 diff。
3. 本周新增或修改的 Obsidian 页面。
4. 数据库模型变化。
5. 权限、多租户、同步日志变化。
6. 飞书、邮件、AI、第三方接口相关改动。
7. 本周测试记录。
8. 本周节点审核记录草稿。

## 审核重点

| 类别 | Claude/Codex 需要检查的问题 |
|---|---|
| 业务范围 | 是否超出一期范围，是否做了客户暂时看不见价值的复杂功能 |
| 权限安全 | 是否存在跨租户数据访问、角色越权、敏感字段泄露 |
| 数据同步 | 是否有来源、状态、失败原因、重试和审计 |
| AI 自动化 | 是否让 AI 自动执行了高风险动作，是否缺人工审核 |
| 飞书/邮件 | 是否同步敏感信息，是否批量发送未经审核内容 |
| 前端体验 | 客户是否能看懂，手机端是否过度复杂 |
| 测试验收 | 是否缺少复现步骤，是否只有口头说明 |
| 交付质量 | 是否有文档、截图、测试记录和风险说明 |

## 触发强制审核的情况

以下情况必须使用 Claude/Codex 做辅助审核：

1. 数据库模型新增或大改。
2. 权限、多租户、用户角色发生变化。
3. AI 输出会影响客户沟通、邮件发送、客服回复。
4. 飞书同步字段包含客户信息、订单信息、联系方式。
5. 邮件自动化开始接触真实客户。
6. 任何第三方平台接入从“测试数据”切换到“真实账号”。
7. 周计划出现重大延期或方向变化。

## 不允许 Claude/Codex 做的事

1. 不直接决定是否给客户上线。
2. 不保存真实密钥、账号、cookie、客户隐私。
3. 不替代人审 拉取请求（PR）。
4. 不用没有来源的数据做商业结论。
5. 不建议绕过平台规则的抓取或自动化操作。

## 项目负责人最终确认

Claude/Codex 输出后，项目负责人必须做人工判断：

- 采纳哪些建议。
- 哪些建议暂不采纳，原因是什么。
- 哪些问题进入 GitHub 任务卡（Issue）。
- 哪些变更写入 Obsidian 决策记录。
