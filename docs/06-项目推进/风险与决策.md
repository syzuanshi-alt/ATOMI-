# 风险与决策

## 当前关键风险

- 平台接口权限审核慢。
- 多租户隔离设计不足。
- AI 输出误执行高风险动作。
- 任务散落在聊天记录里。
- 手机端过早承载复杂后台。
- 飞书被误用为主数据库。
- AI 客服消息中台目前仍是 Demo/模拟接口，没有真实数据库持久化。
- `npm audit` 提示 Next 依赖链中 PostCSS 有 2 个中危风险；当前不执行 `npm audit fix --force`，因为 npm 给出的强制修复会破坏 Next 版本。
- 2026-06-03 新增的 RBAC / Session 准备层改动目前仍未提交，且还没有从现有记录确认完整校验通过。

## 已锁定决策

- GitHub 是代码和任务主控。
- Obsidian 只做知识库。
- 飞书只做业务协同出口。
- 一期先做内部试点。
- 手机端先做 手机网页/可安装网页应用（H5/PWA）。
- AI 工具不允许绕过人工审核和 拉取请求（PR）。
- 后台 UI 可以极简高级，但不能弱化业务状态。风险等级、权限边界、Demo/真实模式、连接状态必须清楚显示。
- 风险等级颜色固定为：低风险绿色、中风险橙色、高风险红色。不要把中风险混用为 Demo 金色。
- Figma 设计稿用于提高视觉质量，但最终以商用后台可判断、可操作、可审计为准。

- PPT 中所有平台接入都按难度分期说明：独立站、表单、邮件、飞书先验证；企业微信、WhatsApp、抖音、TikTok 后续按官方权限接入。
- PPT 中 AI 客服定位为托管助手，只做翻译、总结、草稿、低风险引导和信息收集。
- PPT 中不使用未验证的平台 Logo，不放真实密钥、真实客户资料或真实订单截图。
- 生产交付必须保留多租户、权限、AI 审批、审计日志和同步日志。

## 2026-06-02 客户展示 PPT 新增风险

- 如果向客户演示时把 Demo 能力讲成生产能力，容易造成交付预期失控。
- 如果客户要求直接接个人微信或非官方私信自动化，应拒绝并改走企业微信、官方客服或表单/邮件渠道。
- 如果客户要求 AI 自动承诺退款、补偿、改价或发货时间，应归为高风险需求，必须人工审核。
- 如果 PPT 后续加入真实截图，必须确认截图不包含客户隐私、密钥、订单敏感信息。

## 2026-06-02 凭证与权限护栏新增决策

- 已新增 `GET /api/integrations/credential-readiness`，用于真实接入前检查凭证加密配置是否具备准备条件。
- 凭证预检接口只允许管理员或具备 `integrations.manage` 权限的角色访问，客服角色访问必须 403。
- 凭证预检接口只返回状态，不返回 `SECRETS_ENCRYPTION_KEY`、环境变量原文、平台访问凭证、邮箱密码或明文密钥。
- 当前本地 `SECRETS_ENCRYPTION_KEY` 是占位值，显示“已配置但格式无效”是正确状态。
- 当前仍处于 Demo/沙箱阶段，即使加密配置有效，也不允许保存真实平台访问凭证。
- 数据接入接口已补真实上线前置条件：官方只读接口权限、密钥加密存储、同步日志、失败重试记录、人工审核、操作审计。
- 同步状态已补中文说明，避免小白团队把 `queued` 误解为真实同步完成。
- 本轮在 `codex/credential-readiness` 分支提交：`6cb96c1 feat: add credential readiness guardrails`。

## 2026-06-02 凭证与权限护栏新增风险

- 当前凭证预检只是检查配置状态，不代表已经可以接真实平台。
- 当前没有真实登录 Session 和真实用户授权，`x-demo-role` 仍只能用于 Demo。
- 如果后续团队在未配置有效加密密钥前保存真实凭证，会产生高风险，应禁止。
- 如果后续从 `codex/credential-readiness` 分支切回 `main`，需要注意 main 可能还没有这次凭证护栏提交。
- 下一步应优先补权限矩阵和 API 权限自检，而不是直接接真实平台。

## 2026-06-02 权限矩阵新增风险

## 2026-06-02 真实登录准备新增决策

- 已新增 `GET /api/auth/session-readiness`，用于说明真实登录前置条件和 Demo Header 边界。

## 2026-06-02 真实登录准备新增风险
