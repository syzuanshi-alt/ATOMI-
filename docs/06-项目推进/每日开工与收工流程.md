# 每日开工与收工流程

## 目的

防止项目推进过程中忘记做到哪一步，避免 Codex、Claude Code、人工协作时上下文断掉。

每天固定做两件事：

- 早上开工：先读计划，再动手。
- 下班收工：先同步 Obsidian，再结束。

## 每天早上开工流程

### 第 1 步：读取 Obsidian 总计划

先读：

- [[Codex底层执行原则]]
- [[完整工作计划]]
- [[周计划]]
- [[每日工作推进计划]]
- [[明日计划]]
- [[风险与决策]]
- [[节点审核记录]]

### 第 2 步：确认当前工程状态

检查 Demo 工程：

```text
D:\腕表工程文件\as腕表ai工作流app开发
```

必须确认：

- 当前 Git 分支。
- 当前未提交文件。
- 昨天最后推进到哪里。
- 今天应该先做哪一步。
- 有没有不能碰的文件或风险点。

### 第 3 步：确认今天只做什么

每天开始前只定 1 到 3 个核心任务，不要贪多。

记录到：

```text
D:\SHKF\docs\06-项目推进\每日工作推进计划.md
```

建议格式：

```text
## YYYY-MM-DD 今日计划

- 今日目标：
- 先做：
- 后做：
- 今天不做：
- 风险提醒：
- 验收标准：
```

### 第 4 步：开始执行

执行前必须确认：

- 没有真实密钥。
- 没有真实客户隐私数据。
- 不接个人微信自动化。
- 不让 AI 自动执行高风险动作。
- 不绕过人工审核。

## 每天下班收工流程

### 第 1 步：同步今日完成内容

写入：

```text
D:\SHKF\docs\06-项目推进\每日工作推进计划.md
```

必须写清楚：

- 今天做了什么。
- 改了哪些文件。
- 哪些通过验证。
- 哪些没通过。
- 明天从哪里继续。

### 第 2 步：同步风险和决策

如果今天出现以下情况，必须更新：

```text
D:\SHKF\docs\06-项目推进\风险与决策.md
```

需要更新的情况：

- 发现某个设计不适合商用。
- 发现某个功能会引入安全风险。
- 决定推翻之前的做法。
- 决定某个模块一期不做。
- 决定某个页面必须保留风险颜色、权限状态或审核状态。

### 第 3 步：同步节点审核

如果今天做了阶段性调整，更新：

```text
D:\SHKF\docs\06-项目推进\节点审核记录.md
```

记录：

- 是否跑偏。
- 发现的问题。
- 风险等级。
- 调整策略。
- 下一个节点。

### 第 4 步：同步明日计划

更新：

```text
D:\SHKF\docs\06-项目推进\明日计划.md
```

必须写清楚：

- 明天第一步打开哪个文件。
- 明天先检查哪个页面。
- 明天优先做什么。
- 明天不要做什么。
- 明天是否允许提交 Git。

### 第 5 步：确认 Git 状态

下班前记录：

```text
git status --short
```

如果没有明确要求，不要自动提交。

只有满足以下条件才提交：

- 页面已经人工审查。
- 校验已通过。
- 没有真实密钥。
- 没有真实客户数据。
- Obsidian 已同步。

## 当前固定规则

- Obsidian 是总计划和决策沉淀。
- GitHub 是代码和任务执行主控。
- Codex/Claude Code 必须以最终交付结果为核心，不机械执行不合适的方案。
- Demo / 真实模式不能隐藏。
- AI 高风险动作必须人工审核。
