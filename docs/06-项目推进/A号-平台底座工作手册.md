# A号-平台底座工作手册

## 角色定位

A 号负责平台底座与数据中台。你的任务不是把页面做漂亮，而是保证系统以后能交付给多个客户时不会乱、不会串数据、不会出现权限漏洞。

你每周必须优先回答三个问题：

1. 这个功能会不会导致客户 A 看见客户 B 的数据？
2. 数据从哪里来、同步是否有日志、失败能不能查？
3. 新增表、字段、权限是否已经写进文档和迁移记录？

## 你负责的边界

| 负责 | 不负责 |
|---|---|
| 项目骨架、后端结构、数据库、ORM、权限、多租户、同步日志、审计日志 | 页面视觉美化、飞书表格业务字段最终设计、客户演示话术 |
| 接口（API） 设计、基础测试、数据隔离验证、环境变量安全 | 直接操作第三方平台账号、绕过人工审核 |

## 每周固定动作

| 时间 | 动作 | 结果 |
|---|---|---|
| 周一 | 阅读本周 A 号 任务卡（Issue），确认依赖 | 任务卡（Issue） 评论写明能否开始 |
| 周二 | 完成数据库/接口（API）/权限主工作 | 本地可运行或有设计文档 |
| 周三 | 写测试或验证脚本 | 测试结果截图或命令输出 |

## 第 1 周详细步骤

### A-01 创建项目骨架

1. 读取 `完整工作计划.md`、`一期范围.md`、`总体架构.md`。
2. 确认技术栈：建议 Next.js（网页应用框架） + TypeScript（带类型的 JavaScript） + PostgreSQL（数据库） + Prisma（数据库工具） + Auth.js（登录认证工具）/自建角色权限控制（RBAC）。
3. 建立目录结构草案：
   - `apps/web`：电脑端后台和 手机网页/可安装网页应用（H5/PWA）。
   - `packages/db`：Prisma（数据库工具） 数据库结构文件（schema）、数据库客户端、迁移。
   - `packages/shared`：通用类型、权限枚举、工具函数。
   - `docs`：Obsidian 知识库。
4. 新建 说明文件（README） 中的本地启动说明。
5. 提交 拉取请求（PR），说明项目骨架为什么这样拆。

验收标准：

- 新成员能按 说明文件（README） 拉取项目并理解目录。
- 不出现真实密钥、cookie、token。
- GitHub 有对应 任务卡（Issue） 和 拉取请求（PR）。

### A-02 配置数据库草案

1. 在 `04-数据库模型/一期数据库模型.md` 中确认一期表。
2. 先写数据库草案，不急于接真实业务数据。
3. 优先设计以下实体：
   - tenants
   - users
   - roles
   - permissions
   - user_tenant_roles
   - sync_jobs
   - sync_logs
   - audit_logs
4. 每个表必须有 `tenant_id` 或明确说明为什么没有。
5. 写清楚软删除、创建时间、更新时间、操作人字段。

验收标准：

- 权限和租户模型能支撑多客户。
- 同步日志能记录来源平台、状态、失败原因。
- 数据库文档和 数据库结构文件（schema） 保持一致。

## 第 2 周详细步骤

### A-03 建 tenants/users/roles 基础模型

1. 建立租户、用户、角色、权限基础表。
2. 建立用户和租户之间的关联，不允许用一个全局角色代替所有客户权限。
3. 定义一期角色：
   - 平台总管理员
   - 客户老板/管理者
   - 客户运营
   - 客户客服
   - 只读观察者
4. 写入权限枚举：
   - 数据看板查看
   - 订单查看
   - 客服查看
   - AI 建议审核
   - 飞书同步配置
   - 邮件模板审核
   - 系统设置
5. 生成第一版迁移，并写基础 初始化测试数据（seed） 数据。

验收标准：

- 同一个用户可属于多个租户，但权限独立。
- 默认情况下不能跨租户查询。
- 文档中能解释每个角色能做什么。

### A-04 写租户隔离测试

1. 准备两个测试租户：`demo-cn` 和 `demo-global`。
2. 每个租户创建独立订单、客服会话、AI 日报数据。
3. 写测试验证：
   - 租户 A 查询不到租户 B 订单。
   - 客服角色不能访问系统设置。
   - 只读角色不能触发同步任务。
4. 将测试命令写入 说明文件（README） 或开发文档。

验收标准：

- 至少 3 个权限/租户隔离测试通过。
- 如果测试失败，不允许合并 拉取请求（PR）。

## GitHub 操作步骤

1. 从最新主分支创建分支：`feature/A-编号-简短英文名`。
2. 每完成一个可验证小点就提交一次 commit。
3. 拉取请求（PR） 标题必须带 任务卡（Issue） 编号，例如 `A-03 建 tenants/users/roles 基础模型`。
4. 拉取请求（PR） 描述写清楚：
   - 改了哪些表/接口（API）/权限。
   - 如何验证。
   - 是否影响 B/C。
   - 是否需要 Claude 审核。
5. 合并前确认 B/C 没有被你的字段变更卡住。

## A号小白每日操作细化

### 每天开工

1. 打开 [[每日工作操作模板]]。
2. 打开自己的任务卡，例如 #1、#2、#3、#4。
3. 先看“验收标准”，再开始做。
4. 如果要写代码，先执行 `git -C D:\SHKF pull`，意思是拉取 GitHub 最新内容。
5. 新建分支时，分支名用任务编号，例如 `feature/A-01-project-skeleton`。

### 做数据库和权限时

1. 每新增一个表，先问：有没有 `tenant_id` 或明确的租户隔离说明？
2. 每新增一个权限，先写中文解释：谁能用、谁不能用。
3. 每新增一次同步逻辑，必须有同步状态和失败原因。
4. 不懂技术词先查 [[小白协作术语表]]。

### 提交前

1. 执行 `git -C D:\SHKF status` 查看改了哪些文件。
2. 检查没有真实密钥、客户资料、账号密码。
3. 把验证结果写入任务卡。
4. 发起拉取请求（拉取请求（PR））后，把链接贴回任务卡。
