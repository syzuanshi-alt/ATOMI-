# 小白协作术语表

## 使用原则

这个项目默认用中文工作。除非是平台按钮、代码命令、技术产品名，否则不要单独写英文。

必须保留英文时，统一写成：

```text
英文（中文解释）
```

例如：

- GitHub（代码和任务协作平台）
- Issue（任务卡）
- PR（拉取请求，提交代码给别人审核）
- Next.js（网页应用开发框架）

## GitHub 常用词

| 英文 | 中文解释 | 小白理解 |
|---|---|---|
| GitHub | 代码和任务协作平台 | 项目代码、任务、审核都放这里 |
| Repository / Repo | 仓库 | 项目的总文件夹，线上版本 |
| Issue | 任务卡 | 一件具体要做的事 |
| Project | 项目看板 | 把所有任务卡按状态、负责人、截止日期管理起来 |
| Pull Request / PR | 拉取请求 | 做完改动后，提交给别人检查再合并 |
| Branch | 分支 | 为某个任务临时开的工作线 |
| Commit | 提交记录 | 保存一次改动，类似一个存档点 |
| Push | 推送 | 把本地改动上传到 GitHub |
| Pull | 拉取更新 | 把 GitHub 最新内容下载到本地 |
| Merge | 合并 | 审核通过后把分支改动合到主线 |
| Review | 审核 | 另一个人检查有没有问题 |
| Assignee | 指派人 | GitHub 账号确定后用来指定负责人 |
| Label | 标签 | 给任务分类，如高风险、文档、前端 |
| Milestone | 里程碑 | 一批任务的阶段目标 |

## 看板字段词

| 英文字段 | 中文字段 | 本项目怎么用 |
|---|---|---|
| Status | 状态 | 未开始、进行中、待审核、已完成、阻塞 |
| Owner | 负责人 | A、B、C、项目负责人 |
| Due Date | 截止日期 | 这件事最晚哪天完成 |
| Risk Level | 风险等级 | 低、中、高 |
| Review Gate | 审核关卡 | 谁审核，是否需要 Claude/Codex |
| Phase | 阶段/周次 | 第1周、第2周等 |
| Customer Visible | 客户可见 | 这项成果能不能给客户看 |
| Evidence | 验收证据 | PR 链接、截图、测试结果、文档链接 |
| Blocker | 阻塞原因 | 卡住的具体原因 |

建议在 GitHub 项目看板里直接创建中文字段；如果 GitHub 页面显示英文按钮，按本表理解即可。

| 英文 | 中文解释 | 什么时候必须保留英文 |
|---|---|---|
| Next.js | 网页应用开发框架 | 技术选型、代码目录、任务标题 |
| React | 前端界面开发库 | 技术选型、组件开发 |
| TypeScript | 带类型检查的 JavaScript | 代码语言说明 |
| PostgreSQL | 数据库 | 数据库设计、部署说明 |
| Prisma | 数据库工具 | 数据库模型、迁移、查询 |
| API | 接口 | 平台对接、数据同步 |
| H5/PWA | 手机网页/可安装网页应用 | 手机端形态说明 |
| CLI | 命令行工具 | 本机工具安装、终端操作 |
| README | 说明文件 | 仓库首页、本地启动说明 |
| CODEOWNERS | 代码负责人配置 | 自动指定审核人 |
| Schema | 数据库结构文件 | Prisma 或数据库模型说明 |

| Prompt | 提示词 | AI 日报、AI 审核输入 |
| Token | 密钥/访问令牌 | 只写概念，不要写真实值 |

## 写文档时的规则

1. 普通业务词用中文，不写英文。
2. GitHub 按钮、技术框架、命令名可以保留英文，但第一次出现必须加中文解释。
3. 命令不要翻译，例如 `git push` 不能写成 `git 推送`，只在命令后说明“这是推送到 GitHub”。
4. 客户可见材料尽量少出现英文；内部技术文档可以出现英文，但必须让小白看懂。
5. 所有人写 Issue、PR、节点审核时，先中文说明，再补英文术语。
