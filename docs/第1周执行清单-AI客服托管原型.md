# 第1周执行清单：AI客服托管原型

## 0. 本周目标

时间：2026-06-01 至 2026-06-05。

本周只做“方向确认、中文化准备、AI 客服托管原型准备”，不接真实平台、不发送真实客户消息、不写真实密钥。

本周完成后，团队必须能回答：

1. 当前工程从哪里打开？
2. 完整搭建计划在哪里？
3. Demo 模式和真实模式有什么区别？
4. AI 客服哪些能自动回复，哪些必须转人工？
5. 下一周数据库和权限要补哪些内容？

## 1. 每天固定动作

每天开始前：

1. 打开当前 Demo 文件夹：`D:\腕表工程文件\as腕表ai工作流app开发`。
2. 阅读 `docs\AI电商自动化平台完整搭建执行计划.md`。
3. 阅读当天任务。
4. 确认不会输入真实 token、secret、邮箱密码、客户资料。

每天结束前：

1. 写清楚今天改了什么。
2. 写清楚改了哪些文件。
3. 写清楚怎么打开查看。
4. 写清楚有没有风险。
5. 写清楚明天要做什么。

## 3. 周二：中文化首页和菜单

负责人：B 号主导，A/C 辅助审核。

输入文件：

- `index.html`
- `src\app.js`
- `src\app.ts`
- `components\growth-app.tsx`

输出文件：

- 中文化后的首页和菜单。
- 页面截图或打开说明。

操作步骤：

1. 把页面主标题从腕表增长系统方向改为“AI电商自动化运营平台”。
2. 把英文菜单和岗位名改成中文优先。
3. 必须保留 Demo 标识，避免客户误以为真实接通。
4. 所有英文术语第一次出现时加中文解释。
5. 不改变原有主要页面结构。

验收标准：

- 小白团队能看懂首页。
- 客户能看懂这是电商 AI 自动化平台。
- 页面明确标注 Demo/模拟数据。

不允许做：

- 不接真实接口。
- 不改权限逻辑。
- 不删除原功能入口。

## 4. 周三：做统一客服工作台静态页

负责人：B 号主导，C 号提供客服场景。

输入文件：

- `docs\AI客服消息中台与托管规则.md`
- `components\growth-app.tsx`
- `src\app.ts`
- `src\app.js`

输出文件：

- 统一客服工作台静态页面或静态模块。
- 模拟会话数据。

页面必须包含：

- 渠道：独立站、表单、邮件、飞书通知、企业微信、WhatsApp、抖音、TikTok。
- 会话列表。
- 客户语言。
- 订单号。
- 风险等级。
- 原文。
- 翻译。
- AI 回复草稿。
- 人工接管状态。

验收标准：

- 能看到多渠道消息统一进入一个客服工作台。
- 能区分低风险和高风险消息。
- 能看到 AI 草稿不是直接发送。

不允许做：

- 不真实发送消息。
- 不承诺退款/补偿/发货时间。
- 不接个人微信。

## 5. 周四：做 AI 托管设置和离线日报静态页

负责人：B/C 共同，A 检查后续落库需求。

输入文件：

- `docs\AI客服消息中台与托管规则.md`
- `components\growth-app.tsx`
- `src\app.ts`
- `src\app.js`

输出文件：

- AI 托管设置页面。
- 离线托管日报页面。

AI 托管设置必须包含：

- 托管开关。
- 客服工作时间。
- 允许自动回复类型。
- 禁止自动回复类型。
- 高风险转人工规则。
- 语言翻译开关。

离线托管日报必须包含：

- 昨晚新增会话数。
- AI 自动回复数。
- 待人工处理数。
- 潜在成交客户。
- 售后风险客户。
- 缺少订单号的客户。
- 建议优先级。

验收标准：

- 能演示客服离线后 AI 如何托管。
- 能演示客服上班后如何查看汇报。
- 高风险内容必须转人工。

不允许做：

- 不让 AI 自动处理售后承诺。
- 不让 AI 自动营销群发。
- 不把托管设置写成真实生产能力。
