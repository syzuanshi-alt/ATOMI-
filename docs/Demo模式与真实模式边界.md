# Demo模式与真实模式边界

## 1. 文件目的

本文件用来明确当前项目中“Demo 模式”和“真实模式”的边界，避免团队或客户误解。

Demo 是演示版本，用假数据、假角色、假接口展示业务效果。

真实模式是商用运行版本，必须接入登录、租户、权限、数据库、审计日志和真实平台授权。

## 2. Demo 模式允许什么

Demo 模式允许：

- 使用假客户。
- 使用假订单。
- 使用假达人。
- 使用假广告数据。
- 使用假客服消息。
- 使用假角色切换。
- 使用本地浏览器保存演示输入。
- 使用静态页面展示业务流程。
- 使用模拟接口返回数据。

Demo 模式用途：

- 给客户看效果。
- 给团队确认页面结构。
- 给 Claude Code 理解业务流程。
- 给项目负责人审查范围。

## 3. Demo 模式禁止什么

Demo 模式禁止：

- 输入真实 token。
- 输入真实 secret。
- 输入邮箱密码。
- 输入 WhatsApp、抖音、企业微信等真实平台密钥。
- 输入真实客户姓名、电话、地址、订单号。
- 自动发送真实客户消息。
- 自动发送真实邮件。
- 自动改真实广告预算。
- 自动处理真实退款/补偿。

静态 Demo 里的任何“连接平台”输入框，都只能填假数据。

## 4. 真实模式必须具备什么

真实模式必须具备：

- 登录系统。
- 多租户隔离。
- 用户和角色。
- 权限校验。
- 数据库存储。
- 审计日志。
- 真实平台授权。
- 密钥加密存储。
- AI 输出记录。
- 人工审核记录。
- 错误和同步日志。

如果没有这些能力，就不能称为真实模式。

## 5. x-demo-role 的边界

当前模板中存在 `x-demo-role` 和 `demo_role`。

它们只能用于演示。

原因：

- 用户可以伪造请求头。
- 用户可以修改 cookie。
- 不能证明真实身份。
- 不能用于客户数据权限。

真实模式必须替换为：

- 真实登录会话。
- 用户 ID。
- 租户 ID。
- 角色权限。
- 服务端权限校验。

## 6. 页面数据状态标记

所有页面必须标记数据状态：

| 状态 | 含义 | 能否给客户说是真实接通 |
|---|---|---|
| Demo 数据 | 写死或本地假数据 | 否 |
| 模拟数据 | 按真实结构模拟的数据 | 否 |
| 数据库数据 | 已写入数据库的数据 | 看来源 |
| 真实接口数据 | 从官方接口同步的数据 | 可以，但要说明平台 |

页面上不允许把 Demo 数据说成真实接口数据。

## 7. AI 自动回复边界

Demo 模式：

- 可以演示 AI 回复流程。
- 可以展示 AI 草稿。
- 可以模拟低风险自动回复。
- 不得真的发给客户。

真实模式：

- 低风险自动回复需要客户配置开关。
- 高风险回复必须人工确认。
- 所有 AI 回复必须落库。
- 所有发送动作必须可追溯。

## 8. 平台接入边界

一期优先：

- 独立站在线客服。
- 独立站表单。
- 邮件客服。
- 飞书内部通知。

后续逐步：

- 企业微信/微信客服。
- WhatsApp Business Platform。
- 抖音 IM。
- TikTok/其他社媒。

不做：

- 个人微信自动化。
- 无授权抓取私信。
- 绕过平台规则的自动化。
